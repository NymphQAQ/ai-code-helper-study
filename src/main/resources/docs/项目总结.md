### 技术亮点

1. **完整的 LangChain4j 能力栈集成**  
   项目同时集成了 LangChain4j 的对话模型、流式模型、RAG、Tools、Guardrail 五大核心能力，形成完整的 AI Agent 框架。

2. **SSE + Reactor 流式架构**  
   后端全程响应式，`Flux<String>` 通过 SSE 协议无阻塞地推送到前端，实现打字机效果体验。

3. **RAG 一键增强**  
   将文档放入 `docs/` 目录即可自动纳入知识库，模型回答时自动参考，无需任何额外配置。

4. **Tool Calling 智能调度**  
   `WebSearchTool` 由大模型自主判断是否调用，用户无感知，提供更自然的对话体验。

5. **安全优先设计**  
   在 AI 服务链路的最前端设置 Guardrail，阻止危险请求到达大模型，有效防止提示词注入攻击。

## 架构决策记录

### 为什么选择 LangChain4j 而非直接调用 DashScope SDK？

LangChain4j 提供了统一的抽象层，支持：
- 一键切换不同 AI 提供商（OpenAI、Anthropic、Qwen 等）
- 内置 RAG、Memory、Tools、Guardrail 框架，减少大量样板代码
- `@AiService` 声明式接口，代码更简洁
- 响应式支持（与 Project Reactor 集成）

### 为什么使用内存向量库而非 Milvus/Qdrant？

当前阶段项目规模较小，文档数量有限，内存向量库足够满足需求。  
缺点是每次重启需重新向量化，当文档量大时启动变慢。  
后续可无缝切换为 Milvus（仅需修改 `RagConfig` 中的 `EmbeddingStore` Bean）。

### 为什么会话记忆不持久化到数据库？

当前使用 `ConcurrentHashMap` 的内存方案，实现简单，响应速度快。  
对于个人/小团队使用场景，服务重启时清空历史可以接受。  
生产化时替换为 Redis 实现即可，接口无需改动（只需换掉 `PersistentChatMemoryStore` 的实现）。

### 为什么选择 SSE 而非 WebSocket？

- SSE 为单向推送，符合 AI 对话场景（服务端推数据给客户端）
- SSE 基于 HTTP/1.1，无需额外协议升级，兼容性更好
- 浏览器原生 `EventSource` API 支持，无需第三方库
- 相比 WebSocket 更轻量，实现更简单
